## Part 1 — 사용자 관점

### Canopy 란
게임 개발 프로젝트(**Tactica Defense**)의 기획을 관리하는 **경량 이슈 트래커**. Jira 를 대체한다.

- **왜 만들었나**: Jira 는 계층이 Epic→Task→Sub-task 3단으로 고정이라, 우리 기획 구조(컨텐츠→기능→세부기능→…)를 못 담아 제목에 `[전장-유닛]` 을 욱여넣어야 했다. 그리고 가시성이 나빴다.
- **핵심 가치**: ① 깊은 계층을 **한눈에 보는 가시성** ② 빠르고 단순한 입력 ③ 꼭 필요한 필드만 ④ **요구사항(UR) 추적**.

### 계층 구조
```
프로젝트
 └ 컨텐츠          게임의 큰 영역            예) 전장
    └ 기능          기능 묶음                 예) 소환수
       ├ 세부기능     기능적 동작 (요구사항 소유) 예) 소환수 합성
       └ 마스터데이터  구체 인스턴스             예) 캐릭터 "아리아"
          └ 작업       실제 작업 단위(= 작업내용 묶음)
```
- **세부기능**과 **마스터데이터**는 같은 층. 둘 다 기능의 자식.
- **작업** = 트리의 잎. "작업내용(체크리스트)"의 묶음이고, **도메인 태그**가 붙는다.
- 계층은 **엄격한 타입 문법**으로 강제된다(잘못된 위치엔 노드 생성 불가).

### 요구사항(UR)과 작업 — N:M
- **요구사항(UR)**은 **세부기능이 소유**하는 목록이다(트리 노드 아님).
- 각 UR 은 **수동 상태**를 가진다: **완료 / 미구현 / 오구현** (오구현이면 사유를 기록).
- **작업 ↔ 요구사항은 M:N**: 한 작업이 여러 요구사항을 만족시키고, 한 요구사항을 여러 작업이 만족시킨다(공유 인프라). 작업은 UR 의 하위가 **아니다** — 별개로 존재하고 **링크로 매핑**된다.
- **진행도 = 요구사항(UR) 완료 비율** (작업 완료율이 아님). 마스터데이터는 UR 이 없어 진행도가 "—".

### 작업 도메인 (작업에 붙는 분류)
`기획 / 디자인 / 사운드 / 구현 / 밸런싱 / 기타`

### 상태 · 티켓 키
- **상태**: 모든 노드 타입에 수동 상태. 고정 4 카테고리(**할일 / 진행중 / 완료 / 취소됨**) + 그 안에 커스텀 세부 상태(프로젝트별).
- **티켓 키 = 타입 기반**: `Content-N` / `Feature-N` / `SubFeature-N` / `MasterData-N` / `Task-N`, 요구사항은 `Requirement-N`. (자동 발급)

### 화면 (어떻게 쓰나)
- **트리 뷰**: 좌측 계층 트리(기본 접힘, 리사이즈), 우측 상세. 상세 상단 **커버 헤더**가 레이어(색 구분)와 위치(브레드크럼)를 보여줘 "지금 어디인지" 한눈에. 노드 클릭으로 드릴다운.
- **세부기능 화면**: 요구사항 목록 ↔ 작업 목록 토글. 작업 클릭 → 작업 상세(설명·작업내용·연결 요구사항을 세로로).
- **보드 뷰**: 레이어 선택(작업/세부기능/기능/컨텐츠/마스터데이터) → 상태 칸반, 카드 드래그로 상태 변경.
- **검색**: ⌘K 전역 검색 + 패싯 필터(타입/상태/도메인/담당자/컨텐츠).
- **대시보드**: 요구사항 상태 통계(완료/미구현/오구현) + 미커버 요구사항 + 컨텐츠별 진행률 + 내 작업.

### 현재 데이터
**Tactica Defense** 프로젝트에 검증된 요구사항 분석 결과가 등록돼 있다:
- 컨텐츠 1 · 기능 7 · 세부기능 39 · **요구사항 238** (그룹 없음, 전부 미분류)
- **작업은 아직 비어 있다** → 요구사항을 보고 작업을 채워나가는 단계.

### 기술 스택 (참고)
Vite + React + TypeScript / Tailwind + shadcn/ui (디자인: Pixel Blossom + 메이플스토리체) / Supabase(Postgres) / Vercel. 인증 없음(내부용).

## Part 2 — AI 에이전트 관점 (작업 분석·등록)

### 에이전트의 역할
작업 내용(주로 **요구사항**)을 분석해 → **작업(Task)으로 분해** → **이 프로젝트(Tactica Defense)에 등록(업로드)** 한다.
- 세부기능 밑에 작업 생성, 작업내용(체크리스트) 추가, **작업↔요구사항 링크(M:N)**, 도메인·상태·담당자 지정.
- 전형적 작업: "세부기능의 요구사항을 읽고 → 그걸 만족시킬 작업들로 분해해 등록".

- anon 에 **읽기·쓰기 전체 허용**(내부용). 즉 이 키 하나로 데이터 조작이 다 된다.

### ★ 상세 사용법은 데이터 가이드 참조
**`docs/agent/canopy-data-guide.md`** — 전체 스키마(테이블·컬럼·enum·FK) + REST 호출 예시(GET/POST/PATCH/DELETE) + 자주 쓰는 작업(요구사항 조회 / 작업 생성 / 작업↔UR 링크 / 체크리스트 / 상태 변경 / 트리·진행률 조회). **실제 조작 전에 이 가이드를 읽어라.**

### 반드시 지킬 규칙 (요약 — 자세한 건 가이드)
- **타입 문법 강제**: 컨텐츠→기능→(세부기능|마스터데이터)→작업. 위반 INSERT 는 트리거가 거부한다.
- **티켓 번호 수동 지정 금지**: 트리거가 타입 기반으로 자동 발급(Content/Feature/SubFeature/MasterData/Task, UR=Requirement).
- **요구사항(UR)은 세부기능 소유**: `ur.feature_id` 값은 세부기능 노드의 id 다.
- **작업↔UR 은 `ur_work_link`(M:N)** 로만 연결. 작업을 UR 하위로 만들지 마라.
- **뷰는 읽기 전용**: `node_progress`(UR 완료율), `ur_coverage`(미커버 = 연결 작업 0개 AND status≠완료).
- 무필터 PATCH/DELETE 금지(전체 변경 위험).

### 전형적 등록 흐름
1. Tactica 프로젝트의 세부기능과 그 **요구사항**을 읽는다.
2. 각 요구사항을 만족시킬 **작업**으로 분해한다(한 번에 작업하기 좋은 단위로).
3. 세부기능 밑에 작업을 **생성** → 작업내용(체크리스트) 추가 → 만족하는 **요구사항 링크** → 도메인·상태 지정.
4. 트리·진행률을 조회해 반영을 확인한다.

> 주의: 되돌리기 어려운 쓰기이므로, 등록 전 대상(프로젝트·세부기능·요구사항)을 정확히 식별하고, 중복 생성에 유의한다.

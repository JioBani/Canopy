# 작업 도메인 · 마스터데이터 — 분류 지식 (택티카 방법론 보강)

> 기존 택티카 요구공학 자료([[UR 추출 방법론]]·[[이슈분해-컨벤션]]·[[기획 운영 컨벤션]])에 **빠져 있던 두 분류 지식**을 정리한다.
> - 이슈분해-컨벤션은 "도메인"을 *도메인 언어*(domain language) 뜻으로만 쓰고, **작업 도메인 분류**는 없다. 사실상 **구현(SR)만** Sub-task 로 다룬다.
> - UR 추출 방법론은 "콘텐츠 인스턴스는 UR 이 아니다 → 콘텐츠/밸런스 위임"이라고만 하고, **마스터데이터를 1급 단위로 모델링하지 않는다.**
>
> 이 문서는 Canopy 의 데이터 모델(컨텐츠>기능>세부기능·마스터데이터>작업, [[canopy-data-guide]])과 정합한다. 요구사항 분석·작업 분해 시 이 분류를 적용한다.

---

## 1. 작업 도메인 (Work Domain)

### 정의
UR(사용자 요구사항) 또는 마스터데이터를 **만족·완성시키기 위해 해야 하는 작업**을, **어느 직군/도메인의 일인지**로 분류한 것. Canopy 에서 **작업(Task) 노드에 붙는 태그**다.

### 6개 도메인
| 도메인 | 내용 |
|---|---|
| **기획** | 요소의 **스탯·스킬·컨셉 설계** (캐릭터/스킬의 기획 슬라이스). ※ 캐릭터 원화·스킬 이펙트의 *컨셉/아트*는 기획에 속한다 — '디자인'이 아니다. |
| **디자인** | 요소가 **어떻게 표현·배치되는가** — UI·레이아웃·애니메이션·연출. (컨셉/아트가 아니라 화면 표현) |
| **사운드** | 모든 게임 사운드 (BGM·효과음·보이스) |
| **구현** | 실제 개발 작업. **이 도메인의 작업 요구사항 = 시스템 요구사항(SR)** 이다(별칭). 이슈분해-컨벤션이 다루는 SR 이 곧 **구현 도메인의 작업**이다. |
| **밸런싱** | 기획·콘텐츠 요소의 **수치 밸런싱** (확률·코스트·성장 곡선 등) |
| **기타** | 위에 안 잡히는 것 |

### 핵심 — UR 하나가 여러 도메인으로 fan-out
이슈분해-컨벤션은 UR→**SR(구현)** 만 보지만, 실제로 **하나의 UR 은 여러 도메인의 작업**을 낳는다. 작업은 "도메인 태그가 붙은 작업 요구사항 그룹"이다.

예) UR "플레이어는 소환수 2마리를 합쳐 상위 소환수를 얻는다":
- **기획**: 합성 결과물·규칙 설계
- **디자인**: 합성 UI·연출
- **사운드**: 합성 효과음
- **구현(SR)**: 합성 로직·데이터
- **밸런싱**: 합성 확률·코스트

→ 즉 **"작업"은 구현만이 아니다.** 디자인·사운드·밸런싱 작업도 같은 위치(세부기능/마스터데이터 밑의 작업)에 도메인 태그로 존재한다.

### 스코프 밖 — 상위 활동 (테두리 만들기)
아래 둘은 "요소를 채우는 작업"이 아니라 **기능 구조·세계관의 테두리 자체를 만드는** 활동이라 성격이 다르다. **Canopy 현재 스코프에서는 작업 도메인으로 다루지 않는다**(추후 검토).
- **시스템 기획** — 게임 시스템(메커니즘)의 테두리를 정함 → 기능/세부기능 구조를 낳음.
- **세계관 기획** — 스토리·세계관.

## 2. 마스터데이터 (Master Data)

### 정의
시스템·세계관 기획이 정한 **테두리 안을 "채우는" 구체 인스턴스**. = **컨텐츠 요소**. (캐릭터·몬스터·스테이지·카드, 그 스탯·스킬 등)

UR 추출 방법론이 "콘텐츠 인스턴스는 UR 이 아니다 → 위임"이라며 분석 밖으로 밀어낸 바로 그것을, Canopy 에서 **1급 노드 타입**으로 모델링한 것이다.

### 트리 위치
- **세부기능과 같은 tier** (둘 다 기능의 자식).
- 보통 `소환수 마스터데이터 제작` 같은 평범한 기능 노드 아래에 **잎으로 나열**한다(아리아·고블린…).

### 세부기능 vs 마스터데이터 — 작업 발생 방식이 다르다
| | 세부기능 | 마스터데이터 |
|---|---|---|
| 성격 | 기능적 **동작** ("소환수 합성" — 동사) | 구체 **인스턴스** ("캐릭터 아리아" — 명사) |
| UR | **소유함** → UR 을 낳고, UR 이 도메인 작업으로 fan-out | **없음** — 요소 자체가 작업의 근원 |
| 작업 발생 | 세부기능 → UR → 작업 | 마스터데이터 → **작업 (UR 없이 바로)** |
| 진행도 | 하위 UR 완료율 | UR 없어 UR 기반 진행도 "—" (작업으로 추적) |

### 마스터데이터도 도메인으로 fan-out
마스터데이터 하나도 UR 처럼 여러 도메인 작업을 단다. 예) 마스터데이터 "캐릭터 아리아":
- **기획**: 스탯·스킬·컨셉
- **디자인**: 원화·이펙트·애니메이션
- **사운드**: 보이스·효과음
- **구현**: 데이터 등록·로직
- **밸런싱**: 수치

## 3. 핵심 추상 — UR 과 마스터데이터의 공통점

> **UR 과 마스터데이터는 둘 다 "도메인 묶음(기획·디자인·사운드·구현·밸런싱)으로 fan-out 되는 work-bearing unit"** 이다. 종류만 다르다.
> - UR = 기능적 **요구**(세부기능이 소유)
> - 마스터데이터 = 구체 **인스턴스**

작업을 분해할 때, 이 둘 중 어느 근원에서 나왔든 **"이 요구/요소를 완성하려면 어느 도메인의 무슨 작업이 필요한가"** 를 도메인별로 빠짐없이 길어내는 것이 핵심이다(UR 추출 방법론의 "작업 도출 무중단" 기준을, 구현뿐 아니라 **전 도메인**으로 확장 적용).

## 부록 — 택티카 자료와의 매핑
- 이슈분해-컨벤션의 **SR(시스템 요구사항) = 구현 도메인의 작업 요구사항**. 이 문서의 작업 도메인은 그 SR 을 **디자인·사운드·밸런싱·기획까지 일반화**한 것.
- UR 추출 방법론의 **"콘텐츠 인스턴스(개별 캐릭터 분포 등)는 UR 이 아니다 → 콘텐츠/밸런스 위임"** = 그 위임 대상이 **마스터데이터**. 즉 UR 로 다루지 않기로 한 구체 요소를 Canopy 에서 별도 노드로 추적한다.
- 계층 용어(콘텐츠 > 기능 > 세부기능)는 동일. Canopy 는 그 세부기능 tier 에 **마스터데이터**를 형제로 추가했다.
